Protocol

Speaking HID

The k·no·b·1 communicates over USB HID using 64-byte vendor-specific reports. Inside those packets lives a full JSON-RPC 2.0 layer that controls every aspect of the device — from firmware queries to wallpaper uploads.

01 — Transport Layer

64 Bytes at a Time

All host-to-device communication flows through USB HID reports. The device exposes a vendor-specific HID interface (Usage Page 0xFF00) alongside the standard keyboard and consumer control interfaces. On macOS, the interface is opened in nonExclusive mode to avoid conflicts with the system HID driver.

Each report is exactly 64 bytes. The first three bytes form a header; the remaining 61 bytes carry a UTF-8 payload fragment.

Fragmentation

Messages longer than 61 bytes are fragmented across multiple packets. Each fragment fills the 61-byte payload to capacity (except the last one), and fragments are sent sequentially. The receiving end concatenates payloads until it encounters a complete JSON object or a line ending with \r\n for legacy ASCII commands.

Serial Fallback

A serial UART interface at 115200 baud (8N1) is also available. Over serial, messages are newline-delimited rather than length-prefixed — no 64-byte framing is needed. The same RPC protocols (legacy ASCII and JSON-RPC 2.0) work identically on both transports.

02 — RPC Layer

Two Protocols, One Pipe

The device supports two RPC protocols over the same HID channel. Both coexist and the device disambiguates based on whether the payload starts with # (legacy) or { (JSON-RPC).

Legacy

ASCII Protocol

A simple text-based framing used in early firmware versions and still supported for backward compatibility. 

# Request format:
#METHOD#[PARAMS]#\r\n

# Response format:
#METHOD#RESPONSE#

# Example:
#sys.version##\r\n
#sys.version#0.3.1#
JSON-RPC 2.0

JSON-RPC Protocol

The primary protocol for all modern communication. Follows the JSON-RPC 2.0 specification. 

// Request:
{"jsonrpc":"2.0",
 "method":"sys.version",
 "params":{},
 "id":1}

// Response:
{"jsonrpc":"2.0",
 "result":{"version":"0.3.1"},
 "id":1}

Execution Model

Commands are executed one at a time in a sequential queue. If a second command arrives while the first is still executing, it waits. The host-side timeout is 10 seconds — if no response arrives within that window, the command is considered failed.

Unicode Handling

Non-ASCII characters in JSON payloads are escaped using standard \uXXXX notation. This is particularly relevant for file names and song metadata passed via the media player methods.

04 — Connection Flow

From USB Cable to RPC

Establishing a connection follows a strict sequence. Every step must succeed before the next begins. This is the same flow used by the official Input app and by knob-cli.

1

USB Enumeration

Scan for HID devices with Vendor ID 0x303A (Espressif) and one of the known Product IDs: 0x5555, 0x8296, 0x82E3, and others. The Work Louder node-hid fork adds features specific to this detection step.

2

Filter by Usage Page

Each physical device exposes multiple HID interfaces. Filter for Usage Page 0xFF00 (vendor-defined) to find the RPC interface. Ignore the standard keyboard (0x01/0x06) and consumer control (0x0C/0x01) interfaces.

3

Open HID Interface

Open the vendor-specific HID interface. On macOS, use nonExclusive: true to avoid claiming exclusive access, which would block the system keyboard driver. On Linux and Windows, standard exclusive access works fine.

4

Query Device Status

Send device.status as the first RPC call. The response contains the firmware version, active profile/layer indices, battery level, and charging state. This confirms the RPC channel is alive.

5

Filesystem Sync

Call fs.list with checksum: true to get a manifest of all files on the device. Compare checksums with local copies to determine what needs to be synced (keymaps, wallpapers, lighting configs).

6

Bidirectional RPC Established

The connection is now fully operational. The host can send commands at will, and the device can emit events (media requests, alerts, diagnostic logs) at any time. Both directions share the same 64-byte HID report channel.
