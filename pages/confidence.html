Confidence

What We Know

An honest, evidence-based assessment of our understanding at each level.

Level 1
L1

Protocol Visibility

Confidence
98%

Confirmed

  • Full HID packet format (64 bytes, 0x06 header, channel multiplexing)
  • All 17+ JSON-RPC methods documented
  • Legacy ASCII RPC protocol
  • Transport layer (USB HID + serial fallback)
  • Connection flow and device identification

Unverified

  • Edge cases in fragmentation with very large payloads
  • Error recovery behavior on timeout
  • Exact behavior of undocumented RPC error codes

How we know: Static analysis of Input app v0.14.0 renderer (11 MB) and main process (2.5 MB), community Linux port review, firmware binary strings analysis.

Level 2
L2

Display Control

Confidence
95%

Confirmed

  • Display resolution: 100×310 pixels (k·no·b·1), 170×320 (Nomad E)
  • Image format: LVGL v9, I8 indexed color, 256-entry ARGB8888 palette
  • File size: exactly 32,036 bytes per wallpaper
  • Conversion pipeline: PNG → resize → WASM LVGL converter → chunked upload
  • Full round-trip tested: PNG → LVGL → PNG
  • 11 LVGL widgets compiled into firmware (arc, image, label, chart, menu, dropdown, roller, calendar, textarea, buttonmatrix, span)
  • Full animation system compiled in (lv_anim_start, style transitions, screen transitions)
  • Flex and Grid layout engines both available
  • Rotary encoder registered as LVGL input device (lv_event_get_rotary_diff)
  • 5 firmware screens identified: home, wallpaper, selftest, bootloader, onboarding
  • Pomodoro timer events in firmware (POMODORO_WORK_END, POMODORO_BREAK_END)
  • Custom screens not possible without firmware modification — RPC surface only exposes wallpaper upload and accent color
  • Display controller IC: ST7789 — identified from firmware TFT_eSPI::setRotation() via Ghidra. MADCTL bytes (0x00/0x60/0xC0/0xA0) and per-dimension offsets (e.g. colstart=53, rowstart=40 for 135×240) match TFT_eSPI’s ST7789_Rotation.h exactly; no other controller (ILI9341, GC9A01, ST7735, ST7796, HX8357) fits. Confirmed in both v0.3.1 and v0.3.2-rc.19 firmware builds.

Unverified

  • Display refresh rate and SPI clock speed
  • Whether wallpaper cycling hack (rapid fs.writebin) achieves usable frame rates (~2.5 FPS theoretical)
  • Which LVGL animation paths the firmware actually uses (vs. just compiled in)
  • Whether Pomodoro, Clock, Kebogotchi, and Media Player widgets (all shipped on Nomad E) will be ported to the k·no·b·1 — they’re native C widgets in firmware, so porting requires a firmware release
  • Whether mp.* RPC methods work on the knob (app feature matrix marks mediaPlayer as Nomad E only)

How we know: WASM binary extraction from Input app, format reverse-engineered from binary output analysis across multiple image sizes, app UI crop dimensions (100×310 with aspect ratio 100/310). LVGL widget inventory from firmware binary strings analysis (10,065 lines of extracted strings, cross-referenced with LVGL v9 source paths). App renderer analysis of widget showcase UI (5 slots, 1 functional). Device feature matrix extracted from app config explicitly maps features per device type — the knob gets only alert, wallpaper; Nomad E gets mediaPlayer, alert, wallpaper.

Level 3
L3

Behavioral Extension

Confidence
95%

Confirmed

  • Keymap stored as JSON on device filesystem (keymap.json)
  • Both default keymap JSON blobs extracted from firmware and validated as parseable JSON (2-encoder and 3-encoder variants)
  • Complete top-level schema: version, activeProfileId, profiles[], multiActions[], macros[]
  • Profile schema: name, id, layers[], multiActionsUsed[]
  • Layer schema: id, name, color (24-bit int), layout (encoders / buttons / keymap arrays), os
  • Multi-action schema: id, name, kcOnTap, kcOnHold, kcOnDoubleTap, kcOnTapHold
  • Up to 15 layers × 15 profiles
  • Multi-actions: tap, hold, double-tap, tap-hold
  • Custom keycodes catalogued: BLE (8 slots), layers, profiles, lighting, multi-actions
  • Encoder per-layer CW/CCW/tap configuration verified in extracted JSON

Unverified

  • Full limits of the macro system (how many macros, steps per macro)
  • Maximum keymap.json file size the device can handle
  • Linked app configuration schema (linkedApps)

How we know: Both default keymap JSON blobs extracted from the v0.3.2-rc.19 firmware binary at offsets 0x10735 and 0x1104d. Parsed as JSON, schema documented field-by-field. Keycode vocabulary confirmed by cross-referencing firmware strings with the extracted keymap values.

Level 4
L4

Full Platform Control

Confidence
85%

Confirmed

  • MCU: ESP32-S3 (Xtensa LX7, dual-core, 240 MHz)
  • Display controller IC: ST7789 (Ghidra analysis of TFT_eSPI::setRotation(), verified across two firmware builds — removes a previously-blocking requirement for custom firmware)
  • No secure boot — Secure version: 0 in app descriptor (esptool image-info)
  • No flash encryption — all partitions flags: 0
  • Partition layout fully mapped (bootloader, factory 8 MB, NVS 128 KB, LittleFS 2 MB, coredump 64 KB)
  • Build system: PlatformIO + Arduino-ESP32 + ESP-IDF 5.3.2 (custom build)
  • Project name nomad-e-fw — codebase shared across devices
  • Unstripped ELF exists internally (SHA256 recorded) but not shipped
  • All linked libraries identified (LVGL, TFT_eSPI, NimBLE, TinyUSB, LittleFS, ArduinoJson, mbedTLS)
  • Firmware update via esptool (standard Espressif protocol)
  • Firmware module architecture documented (8 wl-* modules, plus new ones in v0.3.2: wl_charger_controller, wl_rtc, wl_power1, comms_executor)
  • Version diff v0.3.1→v0.3.2-rc.19 reveals active development: host.focused_app RPC, timer controller, joystick support, radial menu, WPM tracking (kb_stats)
  • SDK architecture verified from official docs: app.py runs on the device in MicroPython v1.26.1 with full LVGL (481 bindings) access — lifecycle hooks (start, update, on_event, end), wlsdk module with 5 submodules (ui, EVENT, time, sys, rpc), hot reload with 2s debounce. Optional host-side worker.py runs in Pyodide for HTTP and subprocess. Nomad E only today; codebase shared (nomad-e-fw)
  • SDK source code is not published anywhere public — we only have the binaries. The auto-generated GitHub source archive for sdk-alpha-0.1 contains only a 5-byte README
  • SDK is not secret, just unannounced: custom widgets were a Kickstarter roadmap promise from launch. Product reviews confirm Work Louder’s intent to accept third-party widget submissions. The 🤫 release notes reflect the alpha state, not concealment
  • Nomad E already ships with three native C widgets (Pomodoro, Kebogotchi, Clock) per reviews; these were previously assumed to be preview-only mockups. Corrected via targeted Reddit/press/Instagram search
  • 10 complete Python widget examples captured from the official Examples page — 4 full demos (Rainbow, RPC, BTC Tracker, Sys Stats) plus 6 from the Firmware-Side doc

Unverified

  • SPI pin mapping for TFT display
  • Battery IC model and I2C address
  • Exact LED count and GPIO mapping
  • Whether eFuse vendor data blocks custom firmware
  • Whether SDK firmware will port to k·no·b·1 — and on what timeline

How we know: Firmware binary analysis across 5 versions (v0.2.0 through v0.3.2-rc.19, plus the SDK alpha for Nomad E) — partition table extraction, esptool image-info on split app binary, strings analysis (10,000+ lines per version), ESP-IDF app descriptor parsing, cross-version string diffing to identify newly-added modules and RPC methods, and comparison of stock vs SDK firmware to catalog MicroPython bindings.

Methodology

How Each Finding Was Obtained

1

Electron App Extraction

app.asar from Input v0.14.0 macOS DMG

2

Community Linux Port

worklouder/input-linux GitHub repo

3

WASM Binary Extraction

Base64-encoded 1.7 MB WASM from main process JS

4

Firmware Strings Analysis

v0.3.1 merged.bin (1.4 MB) — strings dump yielded 10,065 lines with C++ symbols, source paths, and LVGL internals

5

GitHub Org Inventory

All 8 repos in the worklouder GitHub org catalogued, including the input-releases-internal experiments repo

6

esptool image-info

App binary split from merged firmware at offset 0x10000. ESP-IDF app descriptor revealed project name nomad-e-fw, secure version 0, compile timestamp, and ELF SHA256

7

Cross-Version String Diff

5 firmware versions (v0.2.0, v0.3.0, v0.3.1, v0.3.2-rc.19, plus Nomad E v0.9.0) extracted and diffed. New modules, RPC methods, and screens identified by version delta

8

SDK Firmware & App Comparison

The SDK alpha firmware for Nomad E (firmware_v0.9.0-sdk.1) was compared string-by-string against stock v0.9.0. The diff revealed 383 new LVGL Python bindings and MicroPython v1.26.1 embedded. The accompanying Input app (input-0.15.0-sdk.1) was then extracted and analyzed for the host-side portion of the architecture.

9

Browser-Automated Doc Capture

Work Louder’s official SDK documentation is hosted on Notion and renders client-side, defeating basic HTTP fetches. We used Playwright + headless Chromium to render, scroll, and extract content from all five sub-pages (main, WL SDK, Input Side, Firmware Side, Examples). The authoritative docs corrected our earlier code-based inference of the SDK architecture — app.py runs on the device, not the host.
