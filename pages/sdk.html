The Architecture

Python, Running on the Knob Itself

The firmware boots an embedded MicroPython runtime with 481 bindings to the full LVGL 9 widget library. Your app.py is loaded from /fs/app.py on the device’s LittleFS, executed in that MicroPython interpreter, and rendered directly to the display. Lifecycle hooks (start, update, on_event, end) run on the device’s frame loop.

An optional worker.py runs in a separate Pyodide sandbox in the Input app on your computer, for tasks the device can’t do easily — HTTP requests, subprocess calls, anything that needs the host. The two communicate over JSON-RPC with a shared wlsdk.* method namespace.

The Data Flow

The firmware loads /fs/app.py on boot and runs it in a MicroPython interpreter with a 20 KiB heap. Your code calls directly into LVGL to build widget trees on the display. A file-watcher hot-reloads the script with a 2-second debounce whenever you save. The worker.py on the host side is optional — you only need it when the device can’t do something itself (typically: HTTP requests or spawning subprocesses). The two sides communicate via JSON-RPC under a shared wlsdk.* method namespace.

Device API

The wlsdk Module — On Device

When app.py runs on the device, you have access to two modules: wlsdk (Work Louder’s SDK surface) and lvgl (the full LVGL 9 widget library, 481 bindings). A single global, lv_root, is injected as the LVGL parent for your UI elements.

Lifecycle Hooks

Your script can define up to four optional top-level functions. The firmware calls them at specific points in the widget lifecycle:

FunctionWhen calledArguments
start()Once, after the script body runsNone
update()Every frame (display refresh loop)None
on_event(event_type, event_index, event_value)On button press/release or encoder turn3 ints
end()On teardown (before reload or screen exit)None

wlsdk Submodules

SubmodulePurposeKey functions
wlsdk.ui Screen behavior & system fonts (stock LVGL Montserrat) get_root(), set_grab_input(), set_block_input(), set_stay_on_screen(), set_disable_overlay(), FONT.BIG, FONT.MEDIUM, FONT.SMALL
wlsdk.EVENT Constants for on_event dispatch BUTTON (0), ENCODER (1), BUTTON_DOWN, BUTTON_UP, ENCODER_RIGHT, ENCODER_LEFT
wlsdk.time Frame timing (for animation) get_delta(), get_elapsed(), get_frames()
wlsdk.sys Platform info get_platform_name() — returns "nomad-v1"
wlsdk.rpc Bidirectional JSON-RPC with host register(method, cb), send_response(ctx, body), send_error(ctx, code, msg), send_notify(method, payload)

Registered RPC methods are automatically namespaced under wlsdk. by the runtime. If you register "ping", the host calls wlsdk.ping. The MicroPython stdlib is available: sys, math, gc, array, struct.

Host API (Optional)

worker.py — For When the Device Can’t

If your widget needs to fetch data from the internet or run shell commands, you write a companion worker.py. It runs in a Pyodide sandbox inside the Input app on the host. Its API is smaller and deliberately host-shaped:

FunctionPurpose
log(msg)Write to the Input app log console
register_notify(method)Subscribe to a notify method triggered from the device
send_rpc(method, params)Send a JSON-RPC message to the device (auto-prefixed wlsdk.)
exec_process(cmd) -> strRun a shell command on the host machine. 10s timeout. Use sparingly.
handle_notify(method, params) you defineRequired callback if you use register_notify. Receives notifies from the device.

Pyodide runs inside Node.js, so you also have access to JavaScript globals via import js. The most useful: await js.fetch(url, options) for HTTP requests. All Python stdlib modules are available.

Example 1

A Centered Clock

The canonical demo — a label centered on the screen, updating every frame. This is the entire app.py. The device runs it, renders it, and updates it at the display’s refresh rate. No host required. 

import lvgl as lv
import wlsdk

label = None

def start():
    global label
    label = lv.label(lv_root)
    label.set_style_text_font(wlsdk.ui.FONT.BIG, 0)
    label.set_style_text_color(lv.color_hex(0xFFFFFF), 0)
    label.center()

def update():
    elapsed = wlsdk.time.get_elapsed()
    mins = int(elapsed) // 60
    secs = int(elapsed) % 60
    label.set_text("{:02d}:{:02d}".format(mins, secs))
Example 2

Responding to Buttons and the Encoder

Call wlsdk.ui.set_grab_input(True) to take input focus away from the firmware, then define on_event. The encoder value accumulates naturally — event_value is +1 for right, -1 for left. 

import lvgl as lv
import wlsdk

status = None
counter = 0

def start():
    global status, counter
    wlsdk.ui.set_grab_input(True)

    cont = lv.obj(lv_root)
    cont.set_size(lv.PCT(100), lv.PCT(100))
    cont.set_flex_flow(lv.FLEX_FLOW.COLUMN)
    cont.set_flex_align(lv.FLEX_ALIGN.CENTER, lv.FLEX_ALIGN.CENTER, lv.FLEX_ALIGN.CENTER)

    status = lv.label(cont)
    status.set_text("Waiting for input...")

def on_event(event_type, event_index, event_value):
    global counter
    if event_type == wlsdk.EVENT.BUTTON:
        if event_value == wlsdk.EVENT.BUTTON_DOWN:
            status.set_text("BTN {} DOWN".format(event_index))
    elif event_type == wlsdk.EVENT.ENCODER:
        counter += event_value
        status.set_text("ENC {} = {}".format(event_index, counter))
Example 3

Two-File Widget — app.py + worker.py

Here’s how the two files cooperate. The device-side app.py sends a notify to the host when the user presses a button. The host-side worker.py receives it, fetches data, and sends the result back via RPC.

app.py — on device

import lvgl as lv
import wlsdk

price_label = None

def start():
    global price_label
    price_label = lv.label(lv_root)
    price_label.set_text("Press to refresh")
    price_label.center()

    wlsdk.rpc.register("write_price", handle_price)
    wlsdk.ui.set_grab_input(True)

def on_event(t, i, v):
    if t == wlsdk.EVENT.BUTTON and v == wlsdk.EVENT.BUTTON_DOWN:
        # Tell the host we need fresh data
        wlsdk.rpc.send_notify("fetch_price", None)

def handle_price(ctx, params):
    price_label.set_text("${:,}".format(params))
    wlsdk.rpc.send_response(ctx, None)

worker.py — on host

import js

register_notify("fetch_price")

async def fetch_price():
    response = await js.fetch("https://api.coinbase.com/v2/prices/BTC-USD/spot")
    data = await response.json()
    send_rpc("write_price", int(float(data.data.amount)))

def handle_notify(method, params):
    if method == "fetch_price":
        fetch_price()

Press the button — the device emits a notify, the host fetches the BTC price, the host calls back into the device with the result, and the device updates the label. Clean, bidirectional, one namespace.

Small API Discrepancy Noted

The Rainbow demo uses wlsdk.event.EVENT_BUTTON (lowercase event submodule with an EVENT_-prefixed constant), while the official docs use wlsdk.EVENT.BUTTON (uppercase, no prefix). One of them is out of date. The other demos use the uppercase wl.EVENT.BUTTON form consistently — suggesting the Rainbow demo is the older one and the docs reflect the current API. Expect API churn: Work Louder’s docs explicitly warn the SDK is experimental.

Capabilities

What This Enables

With Python running on the device and the full LVGL widget library at your fingertips, the surface area of what you can build is much wider than a typical macropad SDK.

Draw Anything

The full LVGL widget library is available: labels, arcs, charts, canvases, flex and grid layouts, gradients, masks, animations. Build any UI the 100×310 (or 170×320) display can show. Colors, fonts, sizes — your choice.

Real Frame Loop

update() runs every frame. Combined with wlsdk.time.get_delta(), this lets you write frame-rate-independent animations, real-time counters, or dashboards that refresh at display rate.

Direct Hardware Input

Buttons and encoder ticks arrive in on_event directly from the firmware. No host in the loop, no round-trip latency. Set grab_input(True) and the device’s physical controls are yours alone.

Hot Reload

Save app.py and the firmware tears down your widget, reloads it from /fs/app.py, and restarts. No reboot, no flashing, no manual reload. Edit-save-see, like a desktop app.

Error Overlay

Uncaught exceptions are caught by the runtime and rendered as a gray overlay with the traceback directly on the device’s display. You can see what broke without plugging into a log. Upload a fix — the overlay clears.

Host Bridge When You Need It

If a widget needs external data, the optional worker.py runs on the host with access to js.fetch, subprocesses, and the full Python stdlib. Only pay the host-dependency cost when you need it.

Constraints

What You Should Know Going In

20 KiB heap

MicroPython gets a 20 KiB heap on the ESP32-S3. Plenty for most widgets, but memory-hungry operations (big buffers, dense data structures) need care. import gc; gc.collect() is your friend.

Small stdlib

On-device stdlib is the MicroPython subset: sys, math, gc, array, struct. If you need HTTP, JSON decoding, regex, or anything heavier, run it in worker.py on the host.

Experimental

Work Louder’s docs open with “This is very much ⚠️ EXPERIMENTAL ⚠️ - APIs and interfaces can and will probably change.” Early alpha means things move. Expect API churn.

The Developer Experience Is Thin

The Input app ships with a CodeMirror Python editor but no example widgets, no starter templates, no embedded API reference. To write a widget you open an empty buffer, look up the Firmware-Side docs on Notion, and start typing. The SDK is remarkably capable for something this raw; the polish will come.

Work Louder’s Stated Roadmap

Custom widgets were part of the Nomad E’s Kickstarter roadmap from launch. Product reviews since then confirm Work Louder’s intent to support third-party widget submissions — suggesting a curated ecosystem model where developers author widgets in Python and Work Louder distributes them. What we’ve captured on this page is the infrastructure for that: the MicroPython runtime, the wlsdk module, the RPC pipe, the hot reload system, the error overlay.

Still not publicly announced: the widget marketplace mechanism, the SDK source-code availability, the timeline for porting to k·no·b·1 or Creator Micro v2.

The SDK exists only for the Nomad E today. Crucially, the SDK is a separate firmware build — stock k·no·b·1 and stock Creator Micro v2 firmware do not ship the MicroPython runtime at all (we verified with strings analysis: zero MicroPython symbols in stock firmwares). The SDK integration (src/sdk/, the custom_widget screen, the MicroPython embed) appears to be modular within the shared nomad-e-fw codebase, so porting to other devices is mostly about display geometry, encoder wiring, and product-specific defaults — not re-implementing the SDK. The host-side SDK app is already device-agnostic.

On Source Availability

The SDK is distributed as binary only. No source code for the SDK firmware, the SDK-enabled Input app, or example widget scripts has been published anywhere public. The firmware references source paths like /src/sdk/wl_sdk.cpp and /src/sdk/mp_json_convert.cpp, confirming the source exists — just not at a URL we can visit. The auto-generated GitHub source archive for the sdk-alpha-0.1 tag contains only a 5-byte README.

Try It Yourself

Getting the SDK

If you have a Nomad E, you can experiment with the SDK today.

  1. Download the SDK firmware and app from the sdk-alpha-0.1 release on GitHub.
  2. Flash firmware_v0.9.0-sdk.1_merged.bin to your Nomad E via the Input app’s bootloader flow, or manually with esptool.
  3. Install the corresponding input-0.15.0-sdk.1 app (macOS arm64, macOS x64, or Windows) — regular Input releases don’t have the Python editor.
  4. Connect the device, open the Custom Widget panel, write Python, upload.

Don’t Flash Nomad E Firmware on a k·no·b·1

Display dimensions, encoder configuration, and pin mappings differ. Flashing Nomad E firmware to a knob may produce a non-functional device. It’s recoverable via the bootloader but not trivial. If you want to experiment with custom firmware on the knob, build from scratch — see Hardware for the platform details.
